# GymOS v3.0 健身房管理員操作手冊 (SOP)

本手冊為 GymOS v3.0 的標準作業流程 (SOP) 指南，旨在幫助管理人員與行政團隊輕鬆維護系統，保障日常營運高效穩定。

---

## 1. 系統權限與職員管理 (Staff Management)

要將一位新進人員設定為「教練」或「管理員」，老闆/主管理員必須直接在 Google 試算表的 `Staff` 分頁進行設定。

### 步驟說明：
1. **獲取真實 LINE ID**：請新進人員先在 LINE 官方帳號內點擊「我的課表」，當作一般學員隨便輸入姓名綁定。這一步是為了讓系統自動抓取他的真實 `line_uid`。
2. **複製 ID**：打開 `Members` (學員表)，找到他剛才綁定的紀錄，複製他的 `line_uid`。
3. **建立職員檔案**：切換到 `Staff` (職員表)，新增一行並填入以下欄位：
   * **`staff_id` (職員編號)**：內部管理的代號，**由你手動自由編碼**（例如教練可編為 `C001`、`C002`，管理員可編為 `A001`）。
   * **`line_uid` (LINE 唯一識別碼)**：貼上剛才複製的那串真實亂碼 ID。
   * **`real_name` (真實姓名)**：該教練/管理員的顯示名稱。
   * **`role` (系統角色)**：
     *   填入 `admin`：擁有最高權限，可進入管理端進行全面維護與排課。
     *   填入 `coach`：僅能進入教練端，進行「出勤校正點名」。
   * **`status` (在職狀態)**：填入 `active` 代表啟用；若員工離職，請改為 `inactive`，系統會立刻撤銷他的所有選單與權限。

> 💡 **提示**：當設定完成後，該人員只要在 LINE 裡面點擊選單，後台會自動偵測其身分升級，並自動呼叫 LINE API 將其選單無縫切換為教練版或管理員版！

## 2. 班級排程與建構 (Class Scheduling)

行政人員在電腦端控制台的「首頁」或「班級經營」分頁，可點選「新增班級」：
1. 輸入班級名稱、選擇班級類型（如團課 1x、2x）、上課難度等級（L1、L2、L3）、指定負責教練及教室。
2. 設定每班上限人數（防爆人數，如 15 人）及開課日期與週數（預設 12 週）。
3. 點擊送出後，系統後台會**在 3 秒內自動在 Google 日曆上建立未來 12 週的所有課堂事件**，並與試算表資料庫完美對齊。

## 3. 教練出勤與點名 (Coach Attendance)

教練點擊 LINE 下方的「📋 教練授課主控」選單，即可進入教練端主頁：
1. **今日課表**：列出該教練今天的所有授課時段。
2. **點名出勤校正**：點擊特定堂數，會即時拉出該堂課已預約上課的所有學員名單。
3. **執行點名**：教練勾選實際出席的學員，點選送出。系統會自動扣減該學員剩餘上課堂數，並將出席狀態標記為「已出席」。

## 4. 一鍵更新 LINE 圖文選單背景圖片 (Rich Menu Synchronization)

為了解決獨立型專案 (Standalone) 無法直接在 Google 試算表選單顯示的問題，我們提供了一套全自動且極度優雅的 **「行動化 / 網頁化一鍵同步」** 機制。你可以在手機或電腦的「管理員控制台」中，隨時更新 LINE 選單圖！

### 第一步：準備精美選單圖片
*   **圖片規格**：請美編製作符合 LINE 官方規定的圖片，尺寸必須剛好是 **`2500 x 843 像素`** (極致黃金矮版比例)，格式為 JPG、JPEG 或 PNG。*(本期專案已為學員端繪製好完美對齊 GymOS 暗黑科技美學的選單圖片，圖檔保存在專案目錄 [c3_rich_menu_member.png](file:///d:/_LINE%20BOT/_C3_PRO/img/c3_rich_menu_member.png) 中)*。
*   **上傳至雲端硬碟**：將圖片上傳至你的個人 Google 雲端硬碟，並複製該圖片的 **「共用/分享連結」**（例如：`https://drive.google.com/file/d/xxxxx/view?usp=sharing`）。**請確保該檔案權限設定為「知道連結的任何人都可以檢視」**。

### 第二步：貼入試算表 Config 設定
*   打開試算表的 **`Config (系統設定)`** 分頁。
*   你會看到最下方自動新增了三個圖片網址專屬欄位：
    *   **`IMG_MENU_MEMBER`**：學員版選單背景圖網址。
    *   **`IMG_MENU_COACH`**：教練版選單背景圖網址。
    *   **`IMG_MENU_ADMIN`**：管理員版選單背景圖網址。
*   請直接將剛才複製的 Google 雲端硬碟共用連結，貼進對應的 **`value`** 欄位中。

### 第三步：一鍵重新同步起飛 (Web App)
1. 在手機 LINE 或電腦瀏覽器中打開 **`管理控制台 (Admin Mode)`**。
2. 看到最上方的控制台卡片中，點擊 **「🖼️ 同步圖文選單」** 按鈕。
3. 網頁會跳出「⏳ 正在向 LINE 同步覆蓋圖文選單...」的提示。
4. **系統底層運作**：系統會自動下載你填寫的雲端硬碟圖片，呼叫 LINE API 重新註冊，並**自動依照本期定案的「方案 B」劃分熱區 (左格：我的課表連結 LIFF / 中格：我要請假 / 右格：我要補課) 綁定**，回填 ID。
5. 靜候大約 5~10 秒，網頁跳出成功提示！此時你與所有學員手機 LINE 下方的選單，就已經無縫完成視覺與熱區功能更新了！

## 5. 一鍵重置資料庫「防誤觸安全鎖」與手動啟用機制

為了防止管理人員在日常營運中「誤觸」重置按鈕導致歷史課表、學員請假補課、與出勤點名紀錄被全數清空，系統預設啟用了「雙重驗證安全鎖」。

### 安全鎖運作機制：
*   **預設防護狀態**：在 **`Config (系統設定)`** 工作表中，設有 **`ALLOW_DATABASE_RESET`** 設定鍵，其設定值預設為 **`false`**。
*   **誤觸保護**：在安全鎖為 `false` 情況下，即使在網頁控制台點擊了「一鍵重置資料庫」按鈕，後端 API 都會**拒絕執行並拋出安全防護警報**，完全保障營運數據的絕對安全！

### 當需要進行重置時的操作步驟 (SOP)：
1.  **開啟安全鎖**：打開試算表的 **`Config (系統設定)`** 分頁，將 **`ALLOW_DATABASE_RESET`** 的設定值 (value) 修改為 **`true`**。
2.  **執行一鍵重置**：打開網頁管理端，在首頁摘要中點擊 **「🗃️ 一鍵重置資料庫與課程種子」**。系統將在 3 秒內重建所有空白分頁並注入初始課程與學員測試資料。
3.  **立刻關閉安全鎖**：重置完成後，請**務必立刻**將試算表中 **`ALLOW_DATABASE_RESET`** 的設定值改回 **`false`**！此舉能百分之百防止日後營運人員再次誤觸。

## 6. 新一期班級一鍵自動續期 (Rollover) 與學費核收指南

本系統提供領先業界的「學員續期 Rollover 引擎」，讓下一期續開班級的流程自動化，大幅減輕行政人員的綁定與登記負擔。

### 步驟 A：一鍵續期與學員自動帶入 (Rollover)
1.  進入管理控制台，切換至 **`🏫 班級管理`** 標籤頁。
2.  找到欲續開新一期的班級藍圖，點擊右側的 **`🔄 續期開班`**。
3.  網頁會跳出「一鍵續期彈窗」：
    *   **新一期季度備註**：輸入此期的標記名稱（例如：`2026年Q3續期`）。
    *   **開始日期**：系統已自動為您推算出前一期結束後的下週開課日，您也可以手動修改。
    *   **勾選舊學員**：彈窗中會自動列出上期所有 active 的活躍學員。預設為全選，您可以手動取消勾選本期不打算續上的學員。
4.  點擊 **`確定一鍵續期並自動轉移`**。
5.  **後台運作機制**：系統會自動計算最大課堂序號，生成全新一期的 Sessions 並同步至 Google 日曆；同時在選課紀錄中將被勾選的學員自動轉移過來，其狀態為 **`pending_payment (待繳費)`**。此時他們的點名冊與日曆已就緒，只待收費！

### 步驟 B：行政收費與學費核點啟用 (Tuition Ledger)
1.  學員繳納學費後，行政人員進入管理控制台的 **`💰 學費核點`** 標籤頁。
2.  這裡會精準呈現所有目前處於 **「待繳學費 ⏳」** 狀態的學員。
3.  確認收到款項後，點擊該學員右側的 **`💰 確認已繳費`** 按鈕。
4.  **後端自動連動**：
    *   自動將學員該班級的選課狀態變更為 **`active`**。
    *   自動依據班級設定的總堂數，一次性填入學員的已繳堂數（點數）。
    *   **主動向該學員的手機 LINE 推送一張高質感的「C3 Fitness 繳費核點收據 🧾」Flex Message 卡片**，讓學員即時收到官方收據，滿意度倍增！

## 7. 智慧請假與補課背後的自動化資料流與日曆對接原理

為了讓管理層與教練團隊清楚了解系統自動化邏輯，以下為「請假」與「補課」背後的秒級資料流與 Google 日曆對接原理：

### 🔄 A. 當學員點選「我要請假」時的後台連動：
1. **狀態標記**：系統自動在試算表 `Sessions` 表中，將該學員該堂課的出席狀態改為 **`leave (已請假)`**。
2. **點數發放**：系統自動在 `Members` 表中，為該學員的補課點數 **`加 1 點`**，寫入保留額度。
3. **Google 日曆即時連動 (重中之重 🌟)**：
   * 系統會即時呼叫 Google Calendar API，在 **1 秒內** 找到對應課程的日曆事件。
   * 自動修改事件的 **「描述 (Description)」** 欄位：**將該請假學員的姓名與 ID 扣除，並更新「已預約上課人數」**。
   * 這能確保教練看手機 Google 日曆時，出席名單 100% 絕對正確，不會對已經請假的人點名！

### ⚡ B. 當學員點選「我要補課」時的後台連動：
1. **點數扣除**：系統自動在 `Members` 表中，扣除該學員 1 點補課點數。
2. **狀態標記**：系統在 `Sessions` 表中，新增一筆該學員在目標課堂的紀錄，狀態標記為 **`makeup (補課)`**。
3. **Google 日曆即時連動 (防爆防超收 🌟)**：
   * 系統會即時呼叫 Google Calendar API，在 **1 秒內** 修改目標課堂的日曆事件。
   * 自動修改日曆事件的 **「描述 (Description)」** 欄位：**將該學員姓名寫入「出席名單」中，並增加已預約人數**。
   * 這能即時鎖定該補課名額，徹底防止實體教室人數超收！

## 8. 課程班級設定規範 (Class Configuration Standards)

### 現行課程班級類型一覽

| 類型 | 課程名稱 | 週堂數 | 計費週期 | 教室 | 人數上限 | 允許補課 |
|------|---------|-------|---------|------|---------|---------|
| **A 類** | 基礎重訓 | 1 次/週 | 12 週固定 (共 12 堂) | RM-02 (小教室) | 8 人 | ✅ |
| **B 類** | 混合重訓 | 2 次/週 | 1 個月動態計算 | RM-01 (大教室) | 15 人 | ✅ |
| **C 類** | 特殊專班 | 1 次/週 | 12 週固定 (共 12 堂) | RM-01 (大教室) | 15 人 | ❌ |

### B 類月費制動態堂數說明

B 類課程 (混合重訓) 採用**月份動態計算**機制，每月實際課堂數不固定：
- 設定方式：`period_type = 'monthly'`，`total_sessions = 0`（由系統計算後覆蓋）
- 計算邏輯：在 GAS 後台執行 `generate()` 時，系統自動找出 `period_start` 所在月份內，設定的 `day_of_week`（如週一+週三）共出現幾天，作為當月總堂數寫回試算表
- 例：2026年5月（31天）週一+週三共出現 9 次 → 當月 9 堂；2026年6月（30天）共出現 9 次 → 下月也可能是 9 堂或 8 堂

> **⚠️ 重要**：B 類月費學員的可補額度 = 請假堂數 - 已登記補課數。月底沒用完的補課額度**不延續**至下月。

### 班級命名規範

命名格式：**`{課程類型} {上課星期}{時段}{班別}`**

| 規則 | 說明 | 範例 |
|------|------|------|
| 女性專班加「女」 | 有性別限制的班級 | `基礎重訓 週三晚女專班` |
| B類多班用A/B/C區分 | 同時段有多班 | `混合重訓 週一三晚A班` |
| 時段簡寫 | 上午/晚 | `週一上午班`、`週一晚班` |

## 9. 課程重置 / 重新開班 SOP (seedClasses Reset)

> **使用時機**：測試階段資料重置、資料修正後需全面重建課程。

### ⚠️ 執行前注意事項
- **重置會清空以下所有資料**：班級設定、課堂紀錄、選課紀錄、請假申請、補課申請、出勤紀錄
- **Google 日曆上的課程事件也會全部自動刪除後重建**
- 正式運行後，**嚴禁執行此重置**，應改以「單班補建」方式操作

### 執行步驟

1. 在本機修改 `src/gas/Setup.ts` 中 `seedClasses()` 的種子資料（班級名稱、日期、設定等）
2. 執行 `npm run deploy` 部署至 GAS
3. 至 GAS 後台 (script.google.com) 選擇函式 **`seedClasses`** 點擊 ▶️ 執行
4. 系統會自動：① 刪除舊日曆事件 → ② 清空試算表資料 → ③ 重新建立班級 → ④ 展開 Sessions → ⑤ 建立日曆事件

### ⚠️ GAS 6 分鐘超時防護
已優化日曆清理邏輯：改用 `calendar.getEvents(開始日期, 結束日期)` 一次批量獲取後比對刪除，避免逐筆呼叫 API 造成超時取消 (Execution cancelled)。

## 10. 正式上線前的開課日期設定 (Go-Live SOP)

目前測試階段所有班級的 `period_start` 統一設為 `2026-05-01` 以便測試。正式上線時，需依照各班實際開課日期進行設定。

### 方法 A：直接修改試算表（日常營運方式）

適用於：正式上線後，各班開課日期已確認且測試資料不需保留。

1. 開啟試算表「**班級設定 (Classes)**」工作表
2. 找到要調整開課日期的班級，直接修改「**本期開課日期**」欄位
3. 同時刪除「**課堂紀錄 (Sessions)**」中對應該班級的舊測試課堂（篩選 `class_id` 刪除對應列）
4. 至 GAS 後台執行 `ClassEngine.generate("班級ID")` 重新展開新的課堂 Sessions 及日曆事件

### 方法 B：從程式碼統一設定後重置（一次性切換正式）

適用於：測試結束、正式上線前需要一鍵切換所有班級日期。

1. 在 `src/gas/Setup.ts` 的 `seedClasses()` 函式頂部，將各班日期集中設定：
```typescript
// 正式上線時，在此統一設定各班開課日期
const START_DATES: Record<string, Date> = {
  'A-MON-1000': new Date('2026-06-01'),  // 基礎重訓 週一上午班
  'A-MON-1900': new Date('2026-06-01'),  // 基礎重訓 週一晚女專班
  // ... 以此類推
  'B-MONWED-1840': new Date('2026-06-01'), // B 類統一月初開始
};
```
2. 執行 `npm run deploy` 部署
3. GAS 後台執行 `seedClasses`，系統自動清理重建

> **提醒**：B 類月費班建議統一在每月 1 日開始，系統會自動計算當月正確堂數。A/C 類 12 週固定班可從任意日期開始，系統從開課日起往後排 12 個符合星期的日期。

## 11. 正式上線後的月度換期 SOP (B類)

每月底，B 類課程進入下一期時，管理員需執行「**月度換期作業**」：

1. **確認當月最後一堂課已上完**
2. **進入管理控制台 → 班級管理**，找到 B 類班級，點選「🔄 續期開班」
3. 輸入下期開始日期（建議填下月 1 日），系統自動動態計算下月實際堂數
4. 勾選本月繼續上課的學員，點選「確定一鍵續期」
5. 系統自動：① 計算下月動態堂數 → ② 產生下月 Sessions → ③ 建立下月 Google 日曆事件 → ④ 建立學員下期 Enrollments (狀態：待繳費)
6. 學員繳費後，在「💰 學費核點」分頁點選「確認已繳費」，系統自動將當月正確堂數填入學員的已繳堂數，並推送 LINE 收據給學員
