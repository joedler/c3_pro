# 健身房會員系統 — LINE 綁定功能規格說明

**文件版本：** v2.0  
**撰寫日期：** 2026-05-19  
**適用對象：** 前端 / 全端開發者（antigravity）  
**文件目的：** 完整描述 LINE 綁定流程的商業邏輯、UI 設計思維、欄位規格，供開發者直接依此編碼

---

## 一、背景與目的

本健身房目前所有學員資料以紙本管理，無電子資料庫。此次開發目標是讓**現有在籍學員**透過 LINE 完成帳號綁定，綁定後即可使用線上請假與補課功能。

同一套綁定流程也適用於**未來新進學員**，因此欄位設計以「從零建檔」為前提，不假設後台已有任何資料。

---

## 二、系統架構概覽

```
LINE Official Account
        │
        ▼
    LIFF 應用程式（本專案主體）
        │
   ┌────┴────┐
   │  綁定流程  │  ◀── 本文件描述範圍
   └────┬────┘
        │
        ▼
   後台資料庫
   ├── 會員資料表
   ├── 課程班級表
   └── 會員-班級綁定表
```

**技術前提（供 antigravity 確認）：**
- 使用 LINE LIFF SDK 取得使用者 LINE userId
- 綁定完成後將 LINE userId 與會員 ID 關聯儲存
- 後台班級資料需事先建好，前端透過 API 取得可選時段清單

## 三、整體流程邏輯

綁定流程共四個步驟，採線性推進，不可跳步：

```
步驟 1           步驟 2           步驟 3           步驟 4
基本資料填寫  →  選擇課程類型  →  選擇上課時段  →  確認送出
（全部自填）    （A / B / C）    （系統篩選後）    （不可逆）
```

**關鍵設計原則：**

1. **選而不填** — 課程相關資訊全部以選擇題呈現，不讓學員輸入任何代碼或文字，杜絕錯誤。
2. **系統篩選可見清單** — 步驟 3 的時段清單，由系統根據步驟 1、2 收集的資料（性別、課程類型）自動過濾後才顯示，學員看不到不適合他的選項。
3. **透明告知篩選依據** — 每一個篩選動作都在畫面上明確說明（例如：「已為你過濾限女時段」），讓學員理解為何清單只有這些選項。
4. **三種班型流程統一** — A、B、C 三類班型皆走相同的四步驟流程，C 類學員直接在步驟 3 選擇可用時段，無需申請審核。

## 四、步驟規格

### 步驟 1：基本資料填寫

**目的：** 建立會員基本檔案，同時取得後續班級篩選所需資訊（性別）。

**設計思維：** 全部欄位皆為空白輸入，無預填資料（因後台目前無電子資料）。五項欄位均為必填，全部填妥才可進入下一步。此設計同時適用於現有學員補建資料及未來新進學員初次建檔。

#### 欄位規格

| 欄位名稱 | 類型 | 必填 | 說明 |
|----------|------|------|------|
| 姓名 | 文字輸入 | 是 | 中文全名，最多 10 字 |
| 性別 | 單選（男 / 女） | 是 | 影響步驟 3 時段篩選邏輯 |
| 出生年月日 | 日期選擇器 | 是 | 格式：YYYY / MM / DD |
| 身高 | 數字輸入（cm） | 是 | 整數，合理範圍 100–250 |
| 體重 | 數字輸入（kg） | 是 | 整數，合理範圍 20–200 |

**欄位排列順序：** 依上表順序由上至下排列，五項連續，無分區標題。

**驗證規則：**
- 姓名不可為空、不可全空白
- 性別必須選擇其中一項
- 出生年月日格式正確即可，不做年齡限制
- 身高、體重須填寫且在合理範圍內；超出範圍顯示 inline 紅字警告並阻擋進入下一步

**進入下一步條件：** 姓名、性別、出生年月日、身高、體重，五項皆填妥且驗證通過，下一步按鈕才啟用。

---

### 步驟 2：選擇課程類型

**目的：** 確認學員目前在上哪一種班型，作為步驟 3 篩選時段的依據。

**設計思維：** 以卡片形式呈現三種班型，每張卡片顯示關鍵摘要資訊，讓學員自行對照認出自己的班。不顯示所有時段細節（那是步驟 3 的事）。

#### 三種班型摘要（顯示於卡片）

| 班型代碼 | 名稱 | 頻率 | 程度 | 人數上限 |
|----------|------|------|------|----------|
| A | 基礎重訓 | 每週一次 | 2–5 級 | 8 人 |
| B | 混合重訓 | 每週兩次 | 6–9 級 | 15 人 |
| C | 特殊專班 | 每週一次 | 2–9 級 | 15 人 |

**三類流程完全一致：** A、B、C 選擇後皆進入步驟 3 選擇時段，無任何班型需要跳轉或走申請流程。

**不確定時的出口：** 卡片下方提供「不確定我在上哪種課」連結，點擊後在頁面內展開簡短說明文字（不跳頁），協助學員判斷。

---

### 步驟 3：選擇上課時段

**目的：** 讓學員從符合條件的時段清單中，選出自己實際在上的那一班。

**設計思維：** 這一步是整個流程最關鍵的防呆機制。系統已知學員性別（步驟 1）與班型（步驟 2），因此能過濾清單，讓學員不可能選到不屬於他的班。

#### 篩選邏輯（後端 API 應實作）

1. 依班型（A / B / C）篩選時段
2. 若學員性別為「男」，過濾掉所有「限女」時段（直接不顯示）
3. 保留所有名額狀態的時段（含額滿），但額滿時段不可選擇

#### 時段卡片顯示欄位

每個時段以卡片呈現，卡片包含：

| 資訊項目 | 範例 | 說明 |
|----------|------|------|
| 上課星期 + 時間 | 週一　晚間 20:00 | 主要識別資訊 |
| 難度等級 | Lv.4 | 協助學員確認是否符合 |
| 剩餘名額 | 剩 3 / 7 人 | 即時名額 |
| 狀態徽章 | 可選 / 快滿 / 額滿 | 見下方規則 |

#### 名額狀態規則

| 剩餘名額比例 | 徽章文字 | 徽章顏色 | 是否可選 |
|--------------|----------|----------|----------|
| > 50% | 可選 | 綠色 | 是 |
| 1–50% | 快滿 | 橙色 | 是 |
| 0 | 額滿 | 紅色 | 否（卡片顯示但 disabled） |

**額滿時段仍顯示的原因：** 學員可能正是那個班的成員，讓他確認「我的班確實存在、只是現在滿了」，避免他誤以為系統資料有誤而重複操作或聯絡老師。

**篩選說明條（banner）：** 清單頂端固定顯示一行說明，例如：
> 已為你篩選：A 基礎重訓・男性可選時段，共 6 個時段

---

### 步驟 4：確認送出

**目的：** 讓學員在送出前完整核對所有資訊，確認無誤後才正式寫入資料庫。

**設計思維：** 這一步是不可逆操作（綁定後名額鎖定），因此必須讓學員有完整的「對帳感」。頁面以摘要清單呈現所有填寫內容，不可直接編輯，提供「返回修改」讓他退回修改。

#### 顯示內容

**個人資料區塊：**
- 姓名
- 性別
- 出生年月日
- 身高 / 體重

**課程資訊區塊：**
- 課程類型（A / B / C）
- 時段（星期 + 時間）
- 難度等級

**名額警示（條件顯示）：**
- 若所選時段為「快滿」狀態，顯示橙色警告條：「此時段剩餘 N 名額，確認後名額立即保留」
- 若在送出的瞬間名額歸零（race condition），回應 409 錯誤並引導重選

**操作按鈕：**
- 主要：「確認綁定」— 送出，不可撤銷
- 次要：「返回修改」— 退回步驟 3（保留所有已填資料）

---

### 綁定完成頁

三種班型皆顯示相同的成功畫面：
- 顯示成功圖示
- 說明文字：「{時段} {課程名稱} 已與你的 LINE 帳號連結」
- 功能提示：「之後可在 LINE 直接請假、查詢補課資訊」
- 按鈕：「前往課表」

## 六、後台資料需求（API 規格草稿）

開發前請確認後台已完成以下建置，前端才能正常運作：

### 需要的 API 端點

```
GET  /api/classes
     Query params: type={A|B|C}, gender={male|female}
     Response: 班級時段清單（含名額資訊）

POST /api/members
     Body: 會員基本資料 + LINE userId
     Response: 新建的 member_id

POST /api/members/{member_id}/bind
     Body: { class_id, line_user_id }
     Response: 200 綁定成功 / 409 名額已滿
```

### 班級清單 API 回傳格式（建議）

```json
[
  {
    "class_id": "A-MON-2000",
    "type": "A",
    "day": "週一",
    "time": "20:00",
    "level": 4,
    "gender_limit": null,
    "capacity": 7,
    "enrolled": 6,
    "allow_makeup": true,
    "status": "open"
  },
  {
    "class_id": "B-MONWED-1840",
    "type": "B",
    "day": "週一 + 週三",
    "time": "18:40",
    "level": 8,
    "gender_limit": null,
    "capacity": 15,
    "enrolled": 14,
    "allow_makeup": true,
    "status": "open"
  },
  {
    "class_id": "A-SAT-1000",
    "type": "A",
    "day": "週六",
    "time": "10:00",
    "level": 2,
    "gender_limit": null,
    "capacity": 8,
    "enrolled": 0,
    "allow_makeup": true,
    "status": "pending"
  }
]
```

`gender_limit` 值：`null`（不限）或 `"female"`（限女）。前端依此欄位決定是否顯示該時段。

`status` 值說明：
- `"open"` — 正常開放，可選
- `"pending"` — 尚未開課，顯示但不可選，卡片標示「即將開課」
- `"full"` — 額滿，可由系統根據 `enrolled >= capacity` 自動判斷，或由後台手動設定

## 七、欄位驗證與錯誤處理

### 前端驗證

| 情境 | 處理方式 |
|------|----------|
| 必填欄位空白 | 下一步按鈕 disabled，不顯示 toast |
| 欄位格式錯誤（如身高輸入文字）| 欄位下方顯示 inline 紅字提示 |
| 身高體重超出合理範圍 | 欄位下方顯示 inline 紅字提示，阻擋進入下一步 |
| 未選課程類型 | 下一步按鈕 disabled |
| 未選時段 | 下一步按鈕 disabled |

### 後端錯誤回應（前端需處理）

| HTTP 狀態碼 | 情境 | 前端顯示 |
|-------------|------|----------|
| 409 | 送出瞬間名額歸零 | 彈出提示：「很抱歉，此時段剛剛額滿，請重新選擇時段」並跳回步驟 3 |
| 422 | LINE userId 已綁定過 | 提示：「你的 LINE 帳號已綁定課程，如需更換請聯繫老師」|
| 500 | 伺服器錯誤 | 提示：「系統發生錯誤，請稍後再試」並保留已填資料 |

## 八、UI 設計規範

整體視覺風格延續 antigravity 現有設計系統，不另行訂定色票與元件規格。以下僅說明本綁定流程特有的互動邏輯與元件行為，供開發時對照實作：

### 進度指示器

頁首固定顯示四格進度條（pill 形狀），狀態分三種：
- 已完成：使用現有系統的成功色（綠色系）
- 進行中：使用現有系統的主色
- 待進行：使用現有系統的邊框淺色

步驟編號文字（「步驟 2 / 4」）顯示於進度條下方，字級 12px。

### 卡片元件（班型卡片、時段卡片）

行為定義：
- 預設狀態：未選中
- 選中狀態：邊框加粗或變色（依現有系統選中樣式）、背景淺色填充
- 不可選狀態（額滿 / 未開課）：整體透明度降低，cursor: not-allowed，點擊無反應

### 按鈕狀態

- 下一步 / 確認綁定：五項資料未填妥或未選擇時維持 disabled 狀態
- 確認綁定送出後立即 disabled 並顯示 loading，防止重複送出

### 資訊提示條（Banner）

兩種語意：
- 說明用（藍色系）：用於步驟 3 頂端說明篩選依據
- 警告用（橙色系）：用於步驟 4 名額緊張提示

### 名額數字顏色（步驟 3 時段卡片）

剩餘名額數字依比例變色：
- 充裕（> 50%）：綠色
- 緊張（1–50%）：橙色
- 額滿（0）：紅色

## 九、狀態管理建議

整個綁定流程的狀態應集中管理（建議使用 React Context 或 Pinia，視技術棧而定），內容如下：

```javascript
{
  // 步驟 1 收集
  name: "",
  gender: null,          // "male" | "female"
  birthdate: "",
  height: null,          // 整數，cm
  weight: null,          // 整數，kg

  // 步驟 2 收集
  classType: null,       // "A" | "B" | "C"

  // 步驟 3 收集
  selectedClassId: null,

  // LINE 資訊（進入 LIFF 時取得）
  lineUserId: "",
  lineDisplayName: "",

  // 流程控制
  currentStep: 1,
  isSubmitting: false
}
```

使用者退回上一步時，所有已填資料必須保留，不得清空。

## 十、其他開發建議

### LINE LIFF 初始化

頁面載入時優先完成 LIFF 初始化並取得 `line.userId`，若取得失敗（例如從一般瀏覽器開啟而非 LINE 內）則顯示提示：「請在 LINE 應用程式中開啟此頁面」並終止流程。

### 離開確認

若使用者在步驟 3 或步驟 4 按下手機返回鍵或關閉 LIFF，顯示確認彈窗：「資料尚未送出，確定要離開嗎？」避免填到一半的資料流失。

### 無障礙

- 所有輸入欄位必須有對應的 `<label>`
- 錯誤訊息使用 `role="alert"` 確保螢幕閱讀器能讀取
- 選中狀態的卡片需有 `aria-selected="true"`
